# ADR-0002：Spec 驱动的文档治理

| 项目 | 内容 |
| --- | --- |
| 状态 | 已采纳 |
| 日期 | 2026-07-15 |
| 关联决策 | [ADR-0001：文档权威来源与同步规则](./ADR-0001-doc-authority.md) |

## 背景

ADR-0001 定义了长期文档的权威来源，但无法管理一次开发变更从提出、实现、验收到归档的生命周期。展站需要避免“只改代码、不回写长期文档”以及“长期 PRD 承载临时实施细节”的问题。

## 决策

文档分为三个层级：

| 层级 | 路径 | 用途 | 生命周期 |
| --- | --- | --- | --- |
| 长期事实 | `docs/` | PRD、ADR、Schema、开发与数据库设计、客户交付文档 | 持续维护 |
| 活跃变更 Spec | `.doc/specs/active/<feature>/` | 单次跨模块功能的目标、方案、任务和验收 | 开发期间 |
| 归档变更 Spec | `.doc/specs/archive/<feature>/` | 已完成变更的决策与验收记录 | 长期保留 |

代码、Zod Schema、Drizzle 定义、SQL migration 和测试属于可执行事实。它们必须与对应的长期文档和活跃 Spec 保持一致，但不被 Markdown 副本替代。

## 变更流程

1. 跨模块或会改变产品、数据、API、配置契约的功能，先从 `.doc/templates/` 创建活跃 Spec。
2. `spec.md` 明确目标、非目标、影响范围与验收标准；`plan.md` 明确技术方案；`tasks.md` 记录可执行任务。
3. 开发期间，活跃 Spec 是本次变更的工作依据；长期 `docs/` 仍是稳定事实的权威来源。
4. 完成验收后，先回写 PRD、Schema、数据库或开发文档中的稳定结论，再将变更包移入 `archive/`。
5. 仅修正文案、链接或不影响语义的排版可直接修改长期文档，不强制创建 Spec。

## 最小同步矩阵

| 变化 | 必须检查或同步 |
| --- | --- |
| SiteConfig 字段 | JSON Schema、Zod、fixture、测试、迁移说明；已持久化时增加 migration |
| 数据库结构 | Drizzle、SQL migration、`DATABASE.md`、API 测试 |
| API 契约 | PRD/API 说明、调用方、接口测试 |
| 产品范围 | PRD、项目计划、活跃 Spec |
| 金源内容或视觉 | 金源 PRD、设计规范、首页文案和素材状态 |

## 后果

- `.doc/` 只承载变更过程，不取代 `docs/`。
- Phase 2 的素材与预览功能开始前，必须创建自己的活跃 Spec。
- CI 成熟后应增加 Markdown 链接检查、Schema/类型一致性测试和 Spec 归档检查。
