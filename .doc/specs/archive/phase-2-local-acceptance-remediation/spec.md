## 背景

Phase 2 归档前的审查发现，`pnpm acceptance:local` 使用手写内存 HTML 和直接读取内存 Map 的方式报告构建 P95 与静态独立性；它没有实际构建模板、没有通过独立 HTTP 服务读取产物，也无法复核文档引用的 Git 忽略占位 Logo。

## 目标

- 本地验收必须对每轮使用真实模板构建产物，避免复用同一 ready Artifact；
- 产物必须由独立静态 HTTP 服务提供，并在不启动 API 网络服务的前提下通过 HTTP 验证 5 个路由、关键 JS 与占位 Logo；
- 中性示例 Logo 纳入版本控制，脚本读取其内容并计算校验和；
- 本地验收报告和长期文档准确描述其范围，不将 HTTP Mock 误称为 HTTPS、云端 P95 或公网独立性。

## 非目标

- 真实 OSS、DNS、TLS、CDN、云端 P95 或微信真机验收；
- 修改生产部署适配器、数据库 schema 或 API 契约；
- 将本地结果表述为真实云交付。

## 影响范围

- API：本地验收命令与其输出；
- 模板：由验收命令进行真实生产构建；
- 素材：增加无客户标识的中性 SVG fixture；
- 文档：修正 Phase 2 归档 Spec、PRD、计划、开发/部署指南和素材清单。

## 验收标准

- [x] 20 轮验收各自构建一个新 Revision/Artifact，并记录每轮实际构建次数；
- [x] 独立静态 HTTP 服务返回首页、产品、资质、关于、联系路由、构建 JS 与示例 Logo；
- [x] 验收过程不启动 API 网络服务，且静态 HTTP 验证在构建/部署任务后执行；
- [x] 示例 Logo 被版本控制，验收脚本读取文件并输出实际 SHA-256 与大小；
- [x] `pnpm acceptance:local`、`pnpm check`、`pnpm build` 通过；
- [x] 长期文档与 Phase 2 归档 Spec 如实更新，随后归档本修复 Spec。

## 验证结果

- 测试：`pnpm check` 通过，23 项测试通过。
- 构建：`pnpm build` 通过。
- 人工验收：`pnpm acceptance:local` 执行 20 次真实模板构建，全部 `healthy`，P95 6134.66 ms，实际构建次数 20；静态 HTTP 服务验证 5 个路由、构建 JS 与版本控制的示例 Logo。
