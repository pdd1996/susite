## 背景

Phase 2 已通过本地/Mock 工程验收，具备不可变 Revision、Asset、BuildArtifact、异步 Deployment、租约 fencing、预览健康检查和静态站独立性。但当前发布器会直接写入站点预览路径，失败任务只记录单个错误摘要，尚未提供有界自动重试、原子切换、历史 artifact 回滚、阶段日志和完整跨站隔离证明。

本阶段按负责人 `Pan` 于 2026-07-16 确认的口径，先用受控本地/Mock 适配器完成可靠性闭环。真实 OSS、DNS、TLS、CDN、云端性能和公网微信真机继续作为独立的部署基础设施启用门槛。

## 目标

- 发布失败时继续提供上一个已确认健康的 artifact，不让半成品覆盖当前预览；
- 对可恢复故障执行有界、幂等、可观测的自动重试，对永久故障快速失败；
- 支持运营人员选择同站点历史 ready artifact 并原子回滚，留下部署与审计记录；
- 持久化 Deployment 每次尝试的阶段日志、错误分类和时间信息，同时过滤密钥及内部敏感信息；
- 以自动化故障注入证明进程中断、租约过期、构建失败、发布失败和健康检查失败时的状态安全；
- 完成跨 `siteId` 读写、部署、日志查询与回滚隔离测试。

## 非目标

- 真实 OSS、preview 公网域名、DNS、TLS 证书、CDN 或云端 P95 验收；
- iOS/Android 微信公网真机验收；
- 正式生产发布、客户域名、DNS 代管、ICP备案或多环境发布；
- 客户自助账号、多角色审批流或 Agent 产品化；
- 无限重试、人工修改历史日志或原地修改 BuildArtifact；
- 将本地/Mock 结果表述为真实云可靠性证据。

## 影响范围

- 产品：后台增加失败详情、重试进度、当前健康版本和历史 artifact 回滚操作。
- 配置契约：不修改 SiteConfig v1。
- 数据库：追加可靠性 migration，持久化当前激活预览、部署类型/尝试调度信息和阶段日志；保留不可变 artifact。
- API：扩展部署状态/日志查询，增加同站点 artifact 列表及回滚接口；所有资源继续执行 actor 与 `siteId` scope 校验。
- Worker / 发布适配器：改为“准备不可变候选版本 → 健康检查 → 原子激活”，增加错误分类、有界退避重试和故障注入点。
- 模板：不改变页面结构；仅作为可靠性验收中的不可变构建输入。
- 测试：增加内存仓库、MySQL 集成、API、Worker、后台交互和本地故障演练。
- 文档：完成后回写 PRD、项目计划、数据库、开发与部署指南。

## 本地 / Mock 验收口径

本阶段可使用受控内存对象存储、独立 loopback 静态 HTTP 服务、可注入失败的发布适配器和独立 MySQL 测试库。Mock 必须执行真实状态机、数据库条件更新、真实模板构建和静态产物读取；不得仅用固定返回值绕过核心发布逻辑。

真实云基础设施仍保持“未启用/待验证”。本阶段通过不解除 IA-01～IA-04，也不授权对外提供公网预览。

## 验收标准

- [x] AC-09：已存在健康版本时，后续构建、发布或健康检查失败均不改变当前激活 artifact；原 URL 仍能读取上一版本的五个路由和关键资源。
- [x] 可恢复错误按稳定分类进入最多 3 次有界重试，使用可测试的退避调度；永久校验/授权/配置错误不重试；进程重启或租约转移不会造成重复激活。
- [x] 每次尝试都有持久化阶段日志，至少覆盖领取、构建、准备发布、健康检查、激活、失败与重试调度；日志包含时间、attempt、稳定错误码，不包含密钥、签名 URL 或原始异常堆栈。
- [x] AC-10：运营人员可选择当前站点的 ready artifact 发起回滚；服务端先验证归属和健康，再原子激活；回滚生成新的 Deployment、日志和审计记录，不修改历史 artifact。
- [x] 回滚或正常发布发生并发竞争时，使用条件更新/fencing 保证只有合法的最新操作激活；过期 Worker 不得覆盖当前版本。
- [x] AC-12：跨站点读取 artifact/Deployment/日志、创建部署、触发回滚或激活均被拒绝，并记录不泄露目标资源细节的审计事件。
- [x] 故障演练覆盖构建异常、对象存储写入异常、健康检查 4xx/5xx、激活前中断、激活后确认丢失、Worker 租约过期和重启恢复。
- [x] 后台能显示当前健康版本、attempt、下一次重试时间、安全错误摘要与阶段日志，并要求显式确认回滚目标。
- [x] MySQL 集成测试在独立 `_test` 数据库执行 migration，验证当前版本原子切换、重试领取、日志归属、跨站回滚拒绝及 fencing。
- [x] `pnpm check`、`pnpm test`、`pnpm build` 和 Phase 3 本地可靠性验收命令通过。
- [x] 长期文档明确区分“本地可靠性已验证”与“真实云基础设施未启用”。

## 验证结果

- 测试：`pnpm check` 通过；共 4 个工作区测试套件通过。`pnpm acceptance:reliability` 通过 8 个 Phase 3/ MySQL 用例，覆盖三阶段失败保护、最多 3 次重试、日志清洗、回滚、跨站拒绝、确认丢失恢复、租约 fencing、并发原子激活和 migration。
- 构建：`pnpm build` 通过，API 类型检查、后台与模板生产构建均成功。
- 本地验收：2026-07-17 复验 `pnpm acceptance:local` 通过 active pointer 连续 20 次真实模板构建，全部 healthy，P95 3103.24 ms，五个路由与关键资源可读；`pnpm acceptance:reliability` 的 AC-09/10/12 故障注入通过。
- 外部门槛：真实 OSS、preview 公网 DNS/TLS/CDN、云端 P95 与公网微信真机仍未执行，不属于本次本地可靠性通过证据。
